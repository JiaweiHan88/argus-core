# Authoring Argus Packs

A **pack** is a versioned, installable unit that extends Argus Core through stable extension
points — without forking or recompiling the app. Core ships with **zero domain content**; everything
domain-specific (log parsers, binary analyzers, evidence typing, domain knowledge, interactive debug
UI, agent tools) is delivered by packs. This guide is the complete contract for authoring one.

> Core reads *all* domain behavior from an in-memory **PackRegistry** populated at startup. If it
> isn't declared in a pack (or one of Core's own generic features), the app doesn't do it.

---

## 1. Anatomy

A pack is a directory whose name **equals its manifest `id`**, containing a manifest plus optional
slot files/dirs:

```
my-pack/                     # dir name MUST equal manifest "id"
  argus-pack.json            # the manifest — the only required file
  persona.md                 # optional: agent persona fragment
  skills/                    # optional: SKILL.md dirs (agent skills)
  references/                # optional: reference docs
  ui/<panel>/                # optional: self-contained web bundles (webPanels)
  bin/                       # optional: prebuilt binaries (published bundle only)
  bin-src/                   # optional: binary SOURCE (never shipped; built by CI)
  CHECKSUMS                  # published bundle only (generated by the build tool)
```

Everything except `argus-pack.json` is optional — a pack can contribute just a persona, just a
detector, just a panel, or any combination.

Two forms of the same pack exist:

- **Source repo** — what you develop in (`bin-src/` holds binary source; no `bin/`/`CHECKSUMS`).
- **Published bundle** — a per-platform `.zip` a user installs (prebuilt `bin/` + `CHECKSUMS`, no
  `bin-src/`). Produced by the build tool (§9).

---

## 2. The manifest — `argus-pack.json`

All identifiers (`id`, binary ids, detector `type`s, window ids, command ids) are **kebab-case**
(`^[a-z0-9]+(-[a-z0-9]+)*$`). The schema uses passthrough, so **unknown keys are preserved, not
rejected** — but Core ignores anything it doesn't recognize.

### Identity (required)

```jsonc
{
  "id": "my-pack",              // kebab; MUST match the directory name
  "displayName": "My Pack",     // human label
  "version": "1.2.0",           // your pack's version (any non-empty string; semver recommended)
  "argusApi": "^1"              // semver RANGE for Core's pack API (see below)
}
```

- **`argusApi`** is a **semver range** that must include the pack API version Core implements
  (currently **`1`**). Core checks `semver.satisfies("1.0.0", argusApi)`. Use `"^1"` to accept any
  Core with pack API 1.x. A pack whose range excludes the running Core is skipped at load with an
  error (dev) / rejected at install (§9).
- **`platform`** (`"<os>-<arch>"`, e.g. `"win-x64"`, `"mac-arm64"`, `"linux-x64"`) is **stamped by
  the build tool** into published bundles and must match the host at install time. Omit it in your
  source manifest — a manifest with no `platform` can be loaded in dev but **cannot be installed**.

### Optional top-level slots

| Key | Type | What it declares |
|---|---|---|
| `persona` | string (path) | A persona fragment file (§3) |
| `binaries` | `Binary[]` | Executables/analyzers Core resolves + exposes (§4) |
| `detectors` | `Detector[]` | Evidence typing + derived-text extraction (§5) |
| `windows` | `Window[]` | webPanel debug UIs + agent-callable commands (§7) |
| `referenceRouting` | `RoutingRule[]` | Keyword→reference routing seeds (§8) |

Plus the directory slots `skills/`, `references/`, `ui/` (§6, §7), which are discovered on disk (not
declared in the manifest, except that `windows[].entry` points into `ui/`).

---

## 3. Persona fragment (`persona.md`)

Point `"persona": "persona.md"` at a Markdown file. Core concatenates its **neutral base persona**,
then every enabled pack's fragment (in **pack-id-sorted order**, so ordering is deterministic across
multiple packs), then any user persona append, into the agent's system prompt.

Use it for role framing and domain file-handling rules the agent must always know — e.g. "these
evidence files are binary; never open them with raw text tools, use the `my-tool` CLI." A missing
file named in `persona` is a load error.

---

## 4. Binaries (`binaries[]`)

Declares executables Core resolves per platform and makes available to detectors (§5) and the agent.
Core never compiles anything — a published bundle ships prebuilt binaries in `bin/`.

```jsonc
{
  "id": "my-tool",                 // kebab; referenced by detectors' extract.bin
  "kind": "exe",                   // "exe" (one executable) | "pathDir" (a dir prepended to PATH)
  "displayName": "My Analyzer",
  "description": "",
  "names": ["my-tool"],            // base name(s); platform .exe handled automatically.
                                   // MUST NOT be git/gh/rm/cd (reserved risk-classified programs)
  "envVar": "MY_TOOL_BIN",         // optional: user env override; also exported to child processes (exe)
  "settingsKey": "myTool",         // optional: key under settings.tools for a user path override
  "devPaths": ["bin-src/target/release"], // dev checkout locations, relative to pack dir;
                                   // "{platformBin}" expands to Scripts (win) / bin (posix)
  "versionArgs": ["--version"],    // exe: prints a version string (health/UI)
  "pathProbeArgs": ["--version"],  // exe: last-resort bare-name-on-PATH probe (must exit 0)
  "doctor": { "cmd": "my-tool", "args": ["doctor"], "json": false }, // pathDir: health/preflight
  "fixHint": "Install my-tool …",  // shown when the binary is missing
  "platforms": ["win32", "darwin", "linux"] // optional: restrict to these process.platform values
}
```

**Resolution order** (first existing hit wins): user `envVar` → user `settingsKey` (`settings.tools`)
→ the pack's `bin/` → `devPaths` → (exe only) a bare-name PATH probe via `pathProbeArgs` → (exe only)
`~/.local/bin`. If nothing resolves, the binary is simply unavailable (extraction that needs it is
skipped, non-fatally).

- **`kind: "exe"`** resolves to a **file path**; if it has an `envVar` and the user didn't set it,
  Core exports the resolved path to that env var for child processes.
- **`kind: "pathDir"`** resolves to a **directory** that Core prepends to `PATH`; detectors then call
  the bare `names[0]`.

**How binaries reach the agent:** the agent runs them as **shell commands via its `Bash` tool**. A
declared binary `name` is auto-classified **LOW risk** (no approval prompt) in Bash command segments.
This is exactly why `git`/`gh`/`rm`/`cd` are forbidden as binary names — they're reserved for Core's
own risk classifier. Tell the agent *how* to invoke your CLIs in your `persona.md`.

## 5. Detectors (`detectors[]`)

Classifies each ingested evidence file to an artifact **`type`**, and optionally extracts **derived
text** so binary evidence becomes searchable/citable.

```jsonc
{
  "type": "widget-trace",          // kebab; the artifact type this detector assigns
  "displayName": "Widget Trace",   // optional; defaults to type
  "analyzeSkill": "analyze-widget-trace", // optional: skill the renderer's Analyze button suggests
  "isText": false,                 // true = the RAW file is text → FTS-index it directly
  "match": [                       // OR of rules; a file matching ANY rule gets this type
    {
      "nameEndsWith": [".wtr", ".wtrace"],   // basename suffix (OR within the array)
      "nameContains": ["widget"],            // basename substring (OR)
      "magicHex": "57545231",                // magic bytes: CONTIGUOUS even-length hex, NO spaces, case-insensitive
      "magicOffset": 0,                      // byte offset for magicHex (default 0)
      "headRegex": { "source": "^WTR", "flags": "" }, // regex over the head as text
      "json": { "anyKeys": ["widgets"], "arrayKeys": ["events"] } // JSON shape checks
    }
  ],
  "extract": { "bin": "my-tool", "args": ["decode", "{input}", "-o", "{output}"] }
}
```

**Matching:** rules within `match[]` are **OR'd**; within a single rule, every *present* field must
match (**AND**). Detectors are evaluated in pack-id-sorted order; the **first** detector that matches
wins. Only the first ~8 KB (the "head") is read for name/magic/regex checks; `json` parses the whole
file (object required). Unmatched files fall back to Core's generics (`archive` / `screenshot` /
`text` / `unknown`). **Duplicate `type` across packs → first declaration wins** (a warning is logged).

**Extraction — the placeholder tokens are `{input}` and `{output}`** (not `{file}`):

- `{input}` → absolute path of the source evidence file.
- `{output}` → `evidence/.derived/<basename>.txt` inside the case dir (created for you).

`extract.bin` references a `binaries[]` **`id`**. Core runs it with the substituted args; the
`{output}` text is FTS-indexed and becomes citable. If the binary is unresolved, extraction is
skipped (non-fatal). Set **`isText: true`** instead when the raw file is already plain text you want
indexed directly (no extraction).

## 6. Skills & references (`skills/`, `references/`)

Drop `SKILL.md` directories under `skills/` and reference docs under `references/`. On startup Core
**seeds** them into the shared stores (`ARGUS_HOME/skills`, `ARGUS_HOME/references`).

- **Skills** seed into the **`bundled`** tier — the **lowest** precedence. A same-named `user` or
  `hivemind` skill **shadows** the pack's. Enabled skills are materialized per case as junctions the
  agent can load. (Precedence, highest→lowest: `user` > `hivemind` > `bundled`.)
- **References** are refreshed from the pack **only when untiered**; a reference a user has promoted to
  a synced tier (`confluence`/`user`/`team-knowledge`/`hivemind`) is never overwritten by a pack copy.

Name skills/refs distinctively (they share a flat namespace with other packs and Core). Uninstalling
the pack reaps its seeded skills and its *untiered* references.

## 7. Windows (`windows[]`) — webPanel debug UI

A pack can ship an interactive debug UI as a **webPanel**: a sandboxed web bundle Core docks in the
case workspace as a tab (pop-out to a floating window supported), which can read case evidence,
contribute to the conversation, and be driven by the agent.

```jsonc
{
  "id": "log-viewer",              // kebab; unique within the pack
  "kind": "webPanel",              // only webPanel today
  "title": "Log Viewer",           // tab label / window title / launcher entry
  "entry": "log-viewer/index.html",// path UNDER ui/ (forward slashes; no .., no absolute)
  "handles": ["widget-trace"],     // artifact type(s) this panel renders → adds an evidence
                                   // "Open in <panel>" action; the opened item arrives via
                                   // getCaseContext().focus (see the bridge below)
  "placement": "tab",              // hint; only "tab" honored (float is user-initiated)
  "network": [],                   // extra origin strings ("https://api.example.com") for the CSP (see below)
  "permissions": ["getCaseContext", "requestEvidence", "readEvidence"], // bridge verbs granted
  "commands": [ /* agent-callable commands — see below */ ]
}
```

If `windows[]` is non-empty, a `ui/` directory **must** exist and every `entry` must resolve to a
real file under it.

### The `ui/<panel>/` bundle — strict CSP

The bundle must be **self-contained**. Core serves it over an `argus-panel://` origin with a strict
CSP: `default-src 'none'; script-src 'self'; style-src 'self'; img-src 'self' data: <network>;
connect-src 'self' <network>; …`.

- **No inline `<script>` or `<style>`, no `eval`** — ship real `.js`/`.css` files referenced with
  `<script src>` / `<link rel="stylesheet">`.
- **No external JS/CSS/fonts** — `script-src`/`style-src`/`font-src` are `'self'` only, even for
  origins you list in `network[]`. `network[]` only widens `img-src` and `connect-src` (e.g. tiles or
  an API the panel fetches). Empty `network[]` ⇒ the panel can reach only its own bundle.
- Assets must stay **within the panel's own subtree** (`ui/<panel>/`).

### Upstream bridge — `window.argus` (opt-in via `permissions[]`)

The panel's only access to Core is a narrow, case+session-bound `window.argus`. Each verb appears
**only if listed in `permissions[]`**:

| Verb | Grants |
|---|---|
| `getCaseContext()` | `{ caseSlug, caseId, sessionId, focus? }` for the bound case |
| `requestEvidence(query)` | FTS search over the case's evidence → hits |
| `readEvidence(id, line?)` | Full text of one evidence item (bound-case only) |
| `cite(relPath, line)` | Drops a citation chip on the chat composer (user sends it) |
| `emitFinding({title, markdown})` | Raises a **MEDIUM, editable approval card**; on approve, writes a finding |
| `sendToAgent(text)` | Stages text into the chat composer for the user to review/send |

Reads are unrestricted (no prompt). `emitFinding` is the only write behind HITL. A panel can never
reach another case/session — the bridge is bound at open time, not by anything the panel supplies.

**How a panel learns which evidence to show:** when it's opened via an evidence "Open in" action (or
the agent's `open_panel` with an evidence id), `getCaseContext()` returns `focus: { evidenceId, line? }`.
The typical flow is `getCaseContext()` → `readEvidence(focus.evidenceId, focus.line)` → render. With
no focus, the panel can offer a `requestEvidence(query)` search to pick an item.

### Downstream commands — `commands[]` → agent tools

Each command becomes an agent tool **`mcp__<packId>__<windowId>_<cmd>`**, registered when the agent
session starts:

```jsonc
{
  "id": "highlight",
  "risk": "low",                   // low = auto-run | medium/high = HITL approval card
  "description": "Scroll the panel to a line and highlight it. Use when the user asks to look at a specific line.",
  "args": ["line"],                // positional arg names; each becomes a string tool param
  "argDescriptions": { "line": "1-based line number to highlight" }
}
```

- **`description`** (falls back to `title`, then a generic string) is what the agent sees to decide
  *when and how* to call the tool — **write a real one**; per-arg `argDescriptions` document each
  parameter.
- **`risk`** drives HITL exactly like every other tool.
- The panel receives dispatched commands via `window.argus.onCommand((cmd, args) => result)`; the
  handler's return value is the agent's tool result. The agent opens a panel with the Core-provided
  **`mcp__argus__open_panel`** tool; a command to a closed panel returns a structured
  `panel-not-open` error (no silent auto-open).

### Reference implementations

Core ships two neutral sample packs you can copy from:

- **`packs/sample-text-viewer/`** — a minimal read-only webPanel (`getCaseContext`/`requestEvidence`/
  `readEvidence`) that renders text evidence with line numbers + find. The smallest complete panel.
- **`packs/sample-bridge-playground/`** — exercises the **full** bridge: every read + write verb
  (`cite`/`emitFinding`/`sendToAgent`) with editable inputs, plus `commands[]` (`highlight`/`echo`)
  and an `onCommand` dispatch log. The reference for write verbs and agent-driven commands.

Both `ui/` bundles are plain self-contained HTML/JS/CSS (no build step) and pass the strict CSP.

## 8. Reference routing (`referenceRouting[]`)

Seeds keyword→reference-file routing rules used by Core's reference-sync feature:

```jsonc
{ "referenceRouting": [ { "keywords": ["widget", "wtr"], "target": "widgets.md" } ] }
```

`target` must be a `.md` basename (a reference file the pack ships).

## 9. Loading, distribution & install

### How Core finds packs

At startup Core loads from **two** directories and merges them (`PackRegistry.load([seeded, installed])`):

- **Seeded** (read-only, Core-shipped): `ARGUS_PACKS_SRC` env override → else `<resources>/packs.seed`
  in a packaged build → else the repo-root `packs/` dir in dev.
- **Installed** (writable): `ARGUS_PACKS_DIR` env override → else **`ARGUS_HOME/packs/`**.

An installed pack of the same `id` **shadows** a seeded one. Load requires: the directory name equals
`manifest.id`, `argusApi` is compatible, the persona file (if named) exists, and every window `entry`
resolves under `ui/`. A pack that fails these is skipped with an error; other packs still load.

**Dev workflow:** point `ARGUS_PACKS_SRC` (or drop your pack in the repo `packs/` dir) at your source
pack — built binaries in place, no packaging — and it loads directly. Changes to the manifest require
an app restart (the registry is built once at startup).

### Building a published bundle

Use the build tool in `tools/pack-tools`:

```
argus-pack build --pack <pack-dir> --bin <prebuilt-bin-dir> --platform <os-arch> --out <dist-dir>
```

It stamps `platform` into the manifest, copies `persona.md` + `skills/`/`references/`/`ui/` + the
`bin/` files, writes a `CHECKSUMS` file (sha256 of every file), and zips it to
`<id>-<version>-<platform>.zip`. Produce one bundle per target platform (a CI matrix); binary source
in `bin-src/` is never shipped.

### Install

Core's Packs settings page installs from a bundle: it verifies `CHECKSUMS` (bidirectionally), checks
`platform` matches the host and `argusApi` is compatible, then atomically swaps it into
`ARGUS_HOME/packs/<id>/`. **Installing/uninstalling always requires an app relaunch** to take effect.
A source manifest with no `platform` cannot be installed (only loaded in place for dev).

## 10. Checklist & common mistakes

- **Directory name ≠ `id`** → the pack won't load. They must be identical.
- **`windows[]` but no `ui/`**, or an `entry` that doesn't exist under `ui/` → load error.
- **Inline `<script>`/`<style>` or external JS/CSS in a panel** → blocked by CSP; ship real files, use
  `'self'` assets, and remember `network[]` only widens `img-src`/`connect-src`.
- **Binary named `git`/`gh`/`rm`/`cd`** → rejected. Pick a distinct name.
- **Detector placeholders** are **`{input}`/`{output}`**, not `{file}`.
- **`extract.bin` must reference a declared `binaries[]` id.**
- **Bland command `description`** → the agent won't know when to call your tool. Describe *when and
  how*, and document each arg.
- **Expecting `argusApi` to be a version** → it's a semver *range* that must include Core's API
  version (`1`); use `"^1"`.
- **Expecting install without `platform`** → build a per-platform bundle; unstamped manifests are
  dev-only.
- **Same-named skill as a user/hivemind skill** → yours (bundled tier) is shadowed. Namespace
  distinctively.

### Minimal valid pack

```jsonc
// my-pack/argus-pack.json
{ "id": "my-pack", "displayName": "My Pack", "version": "0.1.0", "argusApi": "^1",
  "persona": "persona.md",
  "detectors": [ { "type": "widget-trace", "isText": true,
    "match": [ { "nameEndsWith": [".wtr"] } ] } ] }
```

That's a complete, loadable pack: a persona fragment plus a text-evidence detector. Add `binaries`,
`windows`, `skills/`, etc. as your domain needs grow.
